Switching & Bridging

💡 ANALOGY
A switch is like a smart post office that remembers which mailbox belongs to which address — no wasted deliveries to everyone!

Switching and bridging are Layer 2 technologies that intelligently forward frames within a network. Unlike hubs (which broadcast to everyone), switches learn which devices are on which ports and deliver frames only to the intended recipient — dramatically reducing unnecessary network traffic.

A dedicated communication path is established for the entire duration of a session. Used in traditional telephone networks. Resources are reserved end-to-end — even if no data is being sent, the path is held open and unavailable to others.
FORMULA / KEY POINTS
Efficiency = (Actual transmission time) / (Total connection time)
Total delay = Setup delay + Propagation delay + Transmission delay
Resource utilization = Often < 50% (path idle during pauses)
EXAMPLES
1. Traditional PSTN phone calls: when you dial, a circuit is
   established through exchanges all the way to the recipient
   before any voice is even transmitted.

2. ISDN (Integrated Services Digital Network) — digital circuit
   switching used before broadband internet became common.

3. T1/E1 leased lines: dedicated circuits between offices,
   24/30 channels always reserved whether used or not.
Data is broken into packets that travel independently across the network. Each packet may take a different route and arrive out of order. Resources are shared — far more efficient than circuit switching. The Internet is fundamentally a packet-switched network.
FORMULA / KEY POINTS
Total delay = Propagation + Transmission + Queuing + Processing
Store-and-forward: node receives entire packet before forwarding
Cut-through: forward as soon as destination MAC is read

EXAMPLES
1. Email — breaks message into packets, each finds its own route.

2. Web browsing (HTTP/HTTPS) — each HTML, CSS, JS file arrives
   in packets that the browser reassembles.

3. YouTube streaming — video arrives in thousands of IP packets.
   Your device buffers and reorders them for smooth playback.
A bridge connects two LAN segments filtering traffic by MAC address. A switch is a multi-port bridge with hardware ASICs for wire-speed switching. Switches build a MAC Address Table (CAM table) — mapping MAC addresses to physical ports. Unknown destinations get flooded; known ones get unicast-forwarded.
SWITCH LEARNING ALGORITHM
On receiving frame from port P with source MAC M:
  → Add/refresh entry: CAM[M] = P (with timestamp)
  
On forwarding frame to destination MAC D:
  → If CAM[D] exists → forward only to that port
  → If CAM[D] unknown → flood to ALL ports except source
  → If D = FF:FF:FF:FF:FF:FF → flood (broadcast)
EXAMPLES
1. 48-port switch: PC-A (port 3) → PC-B (port 15)
   Switch forwards ONLY to port 15, not all 48 ports.

2. New device plugged in: MAC not in table → switch floods
   first frame, learns MAC from reply, future frames unicast.

3. show mac address-table (Cisco):
   Vlan  Mac Address        Type    Ports
   1     00a1.2b3c.4d5e   DYNAMIC  Gi0/1

Internet Protocol (IP)

💡 ANALOGY
IP is like the postal system's addressing standard — it defines the format of the address on every envelope (packet) traveling through the internet. Without it, nobody would know where to deliver anything.

IP (Internet Protocol) is the principal communications protocol for relaying datagrams (packets) across network boundaries. It provides logical addressing, routing capability, and fragmentation of oversized data. IP is connectionless — each packet is routed independently.

The IPv4 header is 20–60 bytes containing critical routing and control information. Key fields: Version (4 bits), IHL (4 bits — header length in 32-bit words), DSCP/QoS (6 bits), Total Length (16 bits), Identification (16 bits — for fragmentation), Flags (3 bits), Fragment Offset (13 bits), TTL (8 bits), Protocol (8 bits), Header Checksum (16 bits), Source & Destination IP (32 bits each).
KEY CALCULATIONS
Max IPv4 packet size     = 65,535 bytes
Header size (min)        = 20 bytes (IHL=5, 5×4=20)
Data payload             = Total Length − (IHL × 4)
TTL field (time to live) = decremented at each router hop
Protocol field values    = 6 (TCP), 17 (UDP), 1 (ICMP)
EXAMPLES
1. Packet: Src=192.168.1.10, Dst=8.8.8.8, TTL=64, Proto=1(ICMP)
   At each router: TTL decrements. At TTL=0 → discard + send
   ICMP Type 11 (Time Exceeded) back to source.

2. IHL=6 means: header = 6×4 = 24 bytes (has IP options).
   Options used for: timestamps, source routing, record route.

3. Total Length=1500, IHL=5 → payload = 1500-20 = 1480 bytes
   (typical MTU-sized packet on Ethernet)
When a packet is larger than the MTU (Maximum Transmission Unit) of the next network link, IP fragments it. Each fragment carries the same Identification field but has a Fragment Offset showing its position in the original datagram. The DF (Don't Fragment) bit prevents fragmentation. MF (More Fragments) bit is set on all but the last fragment. Reassembly happens only at the final destination.
FRAGMENTATION FORMULAS
Fragment Offset = (start byte position of fragment) / 8
Number of fragments = ⌈Data length / (MTU − Header)⌉

Example: MTU=1500, data=3000 bytes, header=20 bytes
  Max data per fragment = 1500 − 20 = 1480 bytes
  Fragment 1: bytes 0–1479,   offset = 0/8 = 0
  Fragment 2: bytes 1480–2959, offset = 1480/8 = 185
  Fragment 3: bytes 2960–2999, offset = 2960/8 = 370
EXAMPLES
1. Path MTU Discovery (PMTUD): Send DF=1, if "Frag Needed"
   ICMP received, reduce packet size. Used by TCP/QUIC.

2. VPN tunnels reduce effective MTU (overhead added), causing
   fragmentation if not handled → TCP MSS clamping fixes this.

3. IPv6 eliminates router fragmentation entirely — sources must
   discover path MTU and fragment before sending.
IPv4 uses 32-bit addresses providing ~4.3 billion unique addresses — exhausted in 2011. IPv6 uses 128-bit addresses providing 340 undecillion. IPv6 also removes fragmentation at routers, simplifies the header (fixed 40 bytes), builds in IPSec, eliminates broadcast (uses multicast), and uses neighbor discovery instead of ARP.
ADDRESS SPACE
IPv4: 2³²  = 4,294,967,296 addresses
IPv6: 2¹²⁸ = 3.4 × 10³⁸ addresses

IPv4 notation: 192.168.1.1  (dotted decimal, 4 octets)
IPv6 notation: 2001:0db8:85a3::8a2e:0370:7334
  (8 groups of 16-bit hex, :: compresses consecutive zeros)
EXAMPLES
1. Full IPv6:  2001:0db8:0000:0000:0000:0000:0000:0001
   Compressed: 2001:db8::1  (leading zeros + :: for all-zeros)

2. IPv6 loopback: ::1  (equivalent to IPv4's 127.0.0.1)

3. IPv6 link-local: fe80::/10 — auto-configured on every
   interface, only valid on local segment (like 169.254.x.x)

IP Addressing

💡 ANALOGY
An IP address is like a postal address: the network portion is your city/street, and the host portion is your specific house number. Routers are like the regional sorting offices.

IP addresses are 32-bit numbers (IPv4) that uniquely identify interfaces on a network. Written in dotted-decimal notation (e.g., 192.168.1.10), they consist of a network portion (common to all hosts in the network) and a host portion (unique per device). A subnet mask separates the two parts.

Originally IP addresses were divided into five classes determined by the leading bits. Classes A, B, C are for unicast; D for multicast; E is reserved. This system was wasteful — a company needing 1000 hosts had to get a Class B (65,534 hosts), wasting 64,000+ addresses. CIDR replaced classful addressing in 1993.
CLASS RANGES & DEFAULTS
Class A: 1.0.0.0  – 126.255.255.255   /8   16,777,214 hosts
Class B: 128.0.0.0– 191.255.255.255   /16  65,534 hosts
Class C: 192.0.0.0– 223.255.255.255   /24  254 hosts
Class D: 224.0.0.0– 239.255.255.255   Multicast
Class E: 240.0.0.0– 255.255.255.255   Reserved
Note: 127.x.x.x is reserved for loopback (not Class A usable)
EXAMPLES
1. Company A (large): gets 10.0.0.0/8 (Class A) = 16M addresses
   Company B (small): needs 1000 hosts but must get Class B
   = 65,534 addresses. Wastes 64,500+ addresses!

2. Class B leading bits: 10xxxxxx xxxxxxxx . . .
   First two bits "10" → 128.0.0.0 to 191.255.255.255

3. Class D (224.0.0.1): all-routers multicast
   OSPF hello packets sent to 224.0.0.5 (all OSPF routers)
Certain IP ranges are reserved. Private ranges (RFC 1918) are not routed on the public internet — used freely inside organizations with NAT. Link-local (169.254.x.x / APIPA) is auto-assigned when no DHCP is available. The loopback range (127.x.x.x) stays within the local device's IP stack.
RESERVED RANGES
Private (RFC 1918) — not routed on internet:
  10.0.0.0/8       (Class A private) — 16.7M addresses
  172.16.0.0/12    (Class B private) — 1.05M addresses
  192.168.0.0/16   (Class C private) — 65,536 addresses
  
Special:
  127.0.0.0/8      Loopback (127.0.0.1 most common)
  169.254.0.0/16   Link-local / APIPA (no DHCP available)
  0.0.0.0/8        "This network" (unspecified)
  255.255.255.255  Limited broadcast
EXAMPLES
1. Home WiFi: router assigns 192.168.1.x to devices.
   These never appear on public internet — NAT translates them.

2. curl http://127.0.0.1:3000 — connects to local webserver,
   never leaves your machine (loopback interface lo0).

3. APIPA: Windows PC can't reach DHCP server →
   auto-assigns 169.254.x.x, can still talk to other APIPA hosts.
A subnet mask is a 32-bit number with consecutive 1s followed by 0s. The 1s identify the network bits, 0s identify host bits. ANDing an IP address with its mask produces the network address. The broadcast address has all host bits set to 1. Hosts can use addresses between network+1 and broadcast-1.
NETWORK ADDRESS CALCULATION
Network address = IP AND Subnet Mask   (bitwise AND)
Broadcast       = Network OR (NOT Mask) (all host bits = 1)
Usable hosts    = 2^(host_bits) − 2

Example: IP=192.168.10.55, Mask=255.255.255.0 (/24)
  192.168.10.55  → 11000000.10101000.00001010.00110111
  255.255.255.0  → 11111111.11111111.11111111.00000000
  AND result     → 11000000.10101000.00001010.00000000
  Network        = 192.168.10.0
  Broadcast      = 192.168.10.255
  Usable range   = 192.168.10.1 – 192.168.10.254 (254 hosts)
EXAMPLES
1. /30: mask=255.255.255.252, hosts=2^2-2=2 (router links)
2. /16: mask=255.255.0.0, hosts=2^16-2=65,534
3. Is 10.0.0.5 in network 10.0.0.0/24?
   10.0.0.5 AND 255.255.255.0 = 10.0.0.0 ✓ Yes!

CIDR — Classless Inter-Domain Routing

💡 ANALOGY
CIDR is like custom-sized plot allotments — instead of only large, medium, or small fixed sizes, you can carve out exactly the land you need: /30 for 2 hosts, /22 for 1000 hosts.

CIDR (1993) replaced classful addressing and allows flexible allocation of IP blocks. A CIDR block is written as IP/prefix-length (e.g., 192.168.1.0/24). The prefix length indicates how many leading bits define the network — enabling efficient allocation and route aggregation.

The /n prefix specifies network bits. The remaining (32−n) bits are for hosts. Block size doubles for each step down in prefix. ISPs use CIDR to give customers exactly what they need — no more wasteful Class A/B/C allocations. IANA allocates large blocks to RIRs, which allocate to ISPs, which sub-allocate to customers.
BLOCK SIZE FORMULA
Block size    = 2^(32 − prefix_length)
Usable hosts  = 2^(32 − prefix_length) − 2

/24 → 2^8  = 256 addresses, 254 hosts
/23 → 2^9  = 512 addresses, 510 hosts  (2× /24)
/22 → 2^10 = 1024 addresses, 1022 hosts (4× /24)
/16 → 2^16 = 65,536 addresses, 65,534 hosts
/8  → 2^24 = 16,777,216 addresses
EXAMPLES
1. ISP 200.10.0.0/16 → Customer A: 200.10.0.0/24 (256 addrs)
   Customer B: 200.10.1.0/23 (512), Customer C: 200.10.4.0/22

2. Point-to-point link: only need 2 IPs → assign /30
   200.0.0.0/30: .0=network, .1=router1, .2=router2, .3=bcast

3. /26 gives 64 addresses (62 usable). Valid /26 starts:
   .0, .64, .128, .192 (multiples of 64)
CIDR allows combining multiple contiguous blocks into a single routing entry — reducing routing table size. This is called route aggregation or summarization. ISPs aggregate customer routes before advertising to the internet backbone, preventing routing table explosion.
AGGREGATION ALGORITHM
To aggregate n contiguous /k networks into one entry:
1. All networks must share same first (k − log₂n) bits
2. First network address must be divisible by n × 2^(32−k)
3. New prefix = k − log₂n

Example: 200.10.0.0/24, 200.10.1.0/24, 200.10.2.0/24, 200.10.3.0/24
→ 4 networks, n=4, log₂4=2
→ New prefix = 24 − 2 = /22
→ Aggregate: 200.10.0.0/22 ✓
EXAMPLES
1. Can we aggregate 172.16.0.0/24 and 172.16.2.0/24?
   NOT contiguous (gap at 172.16.1.0) → Cannot aggregate!

2. 192.168.0.0/24 + 192.168.1.0/24 → 192.168.0.0/23 ✓
   (starts at .0, divisible by 512 ✓, contiguous ✓)

3. ISP advertising 200.1.0.0/16 to internet covers all 256
   /24 customer blocks — one routing entry instead of 256!
When a router has multiple matching routes for a destination, it chooses the most specific one — the entry with the longest prefix length. This is the fundamental forwarding rule in CIDR-based routing. Without it, aggregation would prevent specific routes from working correctly.
LONGEST PREFIX MATCH RULE
For destination D, select route R where:
  1. D's network bits match R's network/prefix
  2. R has the maximum prefix length of all matching routes

Routing Table:
  0.0.0.0/0    → Router-Internet (default route)
  10.0.0.0/8   → Router-Corp
  10.10.0.0/16 → Router-Dept
  10.10.5.0/24 → Router-Lab

Packet to 10.10.5.55:
  Matches /0, /8, /16, /24 → Choose /24 (longest!)
EXAMPLES
1. Default route 0.0.0.0/0 matches EVERYTHING but is always
   overridden by any more specific route (shortest prefix wins
   only if nothing else matches).

2. Blackhole route: add 10.0.0.0/8 → null0, then advertise
   10.0.1.0/24 → valid next-hop. Traffic to unused space dropped.

3. BGP uses longest prefix match extensively — ISPs inject more
   specific prefixes to influence traffic engineering.

Subnetting

💡 ANALOGY
Subnetting is like dividing a large apartment building (network) into floors (subnets) — residents on the same floor communicate directly; cross-floor communication goes through the building manager (router).

Subnetting divides a single large network into smaller sub-networks, improving security, performance, and address efficiency. Each subnet has its own network address, broadcast address, and range of usable hosts. Devices in different subnets must communicate through a router.

We borrow bits from the host portion of an IP address to create subnet IDs. Each borrowed bit doubles the number of subnets but halves the hosts per subnet. This is a fundamental tradeoff: more subnets = fewer hosts per subnet. The subnet mask is extended by the number of borrowed bits.
SUBNETTING FORMULAS
Subnets created   = 2^s     (s = borrowed bits)
Hosts per subnet  = 2^h − 2 (h = remaining host bits)
New mask          = Original prefix + s bits

Example: 192.168.1.0/24, borrow 3 bits:
  Original host bits = 8, borrow s=3, remaining h=5
  Subnets  = 2^3 = 8
  New mask = /27  (255.255.255.224)
  Hosts    = 2^5 − 2 = 30 per subnet
EXAMPLES
1. 10.0.0.0/8, borrow 8 bits → /16, 256 subnets × 65,534 hosts
2. 172.16.0.0/16, borrow 4 bits → /20, 16 subnets × 4,094 hosts
3. 192.168.5.0/24, borrow 6 bits → /30, 64 subnets × 2 hosts
   (perfect for router point-to-point links!)
The block size is the increment between consecutive subnets. It equals 256 minus the mask value in the interesting octet (the last non-zero octet). List all subnets by repeatedly adding the block size. First address in each block = network, last = broadcast, in-between = usable hosts.
BLOCK SIZE METHOD
Block size = 256 − (interesting octet mask value)
          = 2^(host bits in that octet)

/27 → mask byte = 224 → block = 256−224 = 32
Subnets of 192.168.1.0/27:
  Subnet 0: Net=.0,   Hosts=.1–.30,   Bcast=.31
  Subnet 1: Net=.32,  Hosts=.33–.62,  Bcast=.63
  Subnet 2: Net=.64,  Hosts=.65–.94,  Bcast=.95
  Subnet 3: Net=.96,  Hosts=.97–.126, Bcast=.127
  ... (8 subnets total)
EXAMPLES
1. Is 192.168.1.70 in subnet 192.168.1.64/27?
   .70 > .64 and .70 < .95 (broadcast) → YES ✓

2. /28: block=16, subnets: .0, .16, .32, .48, .64...
   Each has 14 usable hosts.

3. Quick trick: if IP is 172.20.15.200/20, which subnet?
   /20 in 3rd octet: mask=240, block=16. 15/16=0.9...→ base=0
   Network: 172.20.0.0/20, Broadcast: 172.20.15.255
VLSM allows subnets of different sizes within the same address space — maximizing efficiency. Allocate largest subnets first, then carve smaller ones from the remaining space. A /30 wastes no addresses on a 2-router link; a /25 serves a 100-person office.
VLSM ALLOCATION STRATEGY
1. Sort requirements largest to smallest
2. Round each up to nearest power of 2 (+ 2 for net/bcast)
3. Allocate from start of address space, no overlap

Example: 10.0.0.0/24
  Dept A: 60 hosts → needs /26 (62 hosts) → 10.0.0.0/26
  Dept B: 28 hosts → needs /27 (30 hosts) → 10.0.0.64/27
  Dept C: 12 hosts → needs /28 (14 hosts) → 10.0.0.96/28
  Link 1:  2 hosts → needs /30 (2 hosts)  → 10.0.0.112/30
  Link 2:  2 hosts → needs /30 (2 hosts)  → 10.0.0.116/30
  Remaining: 10.0.0.120 – 10.0.0.255 unused
EXAMPLES
1. Without VLSM, all subnets same size → huge waste.
   With VLSM: save 100+ addresses in above example.

2. Enterprise WAN links all use /30 — exactly 2 hosts needed.
   Using /24 for each would waste 252 addresses per link!

3. Used by OSPF, EIGRP, BGP — not supported by RIPv1 (which
   assumes classful masks). RIPv2/OSPF/EIGRP support VLSM.

Supernetting

💡 ANALOGY
Supernetting is like combining multiple small ZIP codes under a single regional postal code for efficient mail sorting — fewer lookup entries for the sorting machine.

Supernetting (route aggregation) combines multiple contiguous networks into a single larger routing entry. This reduces routing table size and is fundamental to how the internet scales. The combined networks must be contiguous, the same size, and the first network's address must be properly aligned.

Supernetting cannot be done arbitrarily. The networks must meet strict criteria: they must be contiguous (no gaps), all the same size (/n), and the starting address must be aligned to the aggregate block boundary. If any condition fails, aggregation is not possible without creating inaccurate routes.
SUPERNETTING REQUIREMENTS
1. All networks same prefix length (/n)
2. Networks are contiguous (no gaps between them)
3. First network address divisible by (count × block_size)
4. New prefix = n − log₂(count)

Check: Can we aggregate 200.1.0.0/24 and 200.1.1.0/24?
  Same size: ✓ (both /24)
  Contiguous: ✓ (.0 and .1 are adjacent)
  Aligned: 200.1.0.0 divisible by 2×256=512? 
    200.1.0.0 in decimal 3rd octet=0 → 0 mod 2 = 0 ✓
  Result: 200.1.0.0/23 ✓
EXAMPLES
1. Can aggregate: 192.168.0.0/24 + 192.168.1.0/24 = /23 ✓
2. Cannot aggregate: 192.168.1.0/24 + 192.168.2.0/24
   (not aligned — 192.168.1.0 not on /23 boundary!)
3. Can aggregate 4: 10.0.0.0/24 through 10.0.3.0/24 = /22 ✓
   (10.0.0.0 divisible by 4×256=1024 in 3rd octet: 0 mod 4=0)
To find the aggregate prefix: write the first network address in binary, zero out the last log₂(count) bits of the network portion, and reduce the prefix by log₂(count). This gives the new summarized network. The aggregate covers exactly those networks and no others (if properly aligned).
STEP-BY-STEP ALGORITHM
Aggregate 4 × /24: 10.0.0.0, 10.0.1.0, 10.0.2.0, 10.0.3.0

Step 1: First network in binary
  10.0.0.0 = 00001010.00000000.00000000.00000000

Step 2: Current prefix = 24, count = 4, log₂4 = 2
  New prefix = 24 − 2 = 22

Step 3: Zero last 2 network bits (bits 23-24)
  First 22 bits = 00001010.00000000.000000xx → 00001010.00000000.00000000
  → 10.0.0.0/22 ✓

Step 4: Verify coverage:
  10.0.0.0/22 covers 10.0.0.0 – 10.0.3.255 ✓
EXAMPLES
1. 8 × /24: 172.16.0.0–172.16.7.0 → 172.16.0.0/21
   (24 − log₂8 = 24 − 3 = 21)

2. 2 × /16: 10.0.0.0/16 + 10.1.0.0/16 → 10.0.0.0/15

3. ISP with 256 customer /24 blocks (200.1.0.0–200.1.255.0)
   → Advertises 200.1.0.0/16 to internet (saves 255 routes!)
Supernetting is the backbone of internet scalability. The global BGP routing table would have millions of entries without aggregation; with it, it's ~1 million. However, aggregation hides individual network states — if one subnet fails, the aggregate may still look reachable, potentially causing a "black hole" (traffic dropped silently).
IMPACT METRICS
Route table reduction = (n − 1) entries saved per aggregation
Global BGP table: ~900K prefixes (without aggregation: millions)
Memory saved: ~100 bytes × routes × routers

Black-hole prevention:
  → Use null0 route for aggregate on originating router
  → aggregate-address (BGP) with summary-only keyword
  → Only advertise aggregate if at least one component active
EXAMPLES
1. ISP aggregates 256 /24s to one /16 → every internet core
   router needs 1 entry instead of 256. Scales globally!

2. Black hole risk: 192.168.0.0/22 aggregated, but
   192.168.2.0/24 goes down. Aggregate still advertised →
   traffic to .2.x routed to aggregate owner → dropped!

3. Solution: Cisco BGP config:
   aggregate-address 192.168.0.0 255.255.252.0 summary-only

Network Address Translation (NAT)

💡 ANALOGY
NAT is like a company receptionist — all calls from employees go out through one company number. The receptionist remembers who made each call and correctly routes back incoming replies.

NAT allows multiple devices on a private network to share a single public IP address. The NAT device (typically a router) maintains a translation table mapping private IP:port pairs to public IP:port pairs, enabling outbound connectivity for all private hosts through a single public address.

When a private host sends a packet, the NAT router replaces the source private IP with its public IP and records the mapping. Incoming replies are matched against the table and forwarded. Static NAT = fixed 1:1 mapping. Dynamic NAT = pool of public IPs. PAT (Port Address Translation) = many-to-one using port numbers.
NAT TYPES
Static NAT:  Private IP ↔ Public IP (1-to-1, permanent)
             Used for: internal servers, permanent access
Dynamic NAT: Private IP ↔ Pool of public IPs (temporary)
             Connection-based, pool must have enough IPs
PAT / NAPT:  Many private IPs → One public IP (port-based)
             Table: (Private IP, PvtPort) ↔ (Public IP, PubPort)
             Supports 65,535 simultaneous connections per IP!
EXAMPLES
1. Home router (PAT): phone(.10:5000), laptop(.11:6000),
   TV(.12:7000) all share 1 public IP via different ports.

2. Static NAT: company web server 192.168.1.100 always maps
   to 203.0.113.50 → internet users reach it at .50.

3. NAT table entry: 
   192.168.1.10:1024 ↔ 203.0.113.1:10500 (TCP, state:ESTAB)
PAT (also called IP Masquerade on Linux, or NAPT) uses port number remapping to distinguish multiple connections sharing one public IP. The router allocates a unique source port for each outbound connection. On reply, the port number in the translation table identifies which private host to forward to.
PAT PORT REMAPPING
Multiple hosts, same dest port → unique public ports:
  10.0.0.1:4500 → 203.0.113.1:10001  (to Netflix :443)
  10.0.0.2:4500 → 203.0.113.1:10002  (to Netflix :443)
  10.0.0.3:4500 → 203.0.113.1:10003  (to Netflix :443)

Reply to :10002 → router knows → forward to 10.0.0.2:4500
Reply to :10001 → router knows → forward to 10.0.0.1:4500
EXAMPLES
1. iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE
   (Linux PAT — masquerade all outbound traffic)

2. Cisco IOS:
   ip nat inside source list 1 interface Fa0/0 overload
   (overload keyword = PAT)

3. Theoretical max: 65,535 ports per protocol per public IP
   (~65K TCP + 65K UDP simultaneous sessions)
NAT breaks the internet's end-to-end principle. Protocols embedding IPs in their payload (FTP active mode, SIP/VoIP) break through NAT. Peer-to-peer (BitTorrent, WebRTC) struggles since NAT'd peers can't accept inbound connections. Solutions: STUN, TURN, ICE, UPnP.
NAT TRAVERSAL TECHNIQUES
STUN  (Session Traversal Utilities for NAT):
  → Client asks STUN server for its public IP:port
  → Used by WebRTC, VoIP to discover external address
  
TURN  (Traversal Using Relays around NAT):
  → Relay server forwards traffic when direct fails
  → Higher latency, always works
  
ICE   (Interactive Connectivity Establishment):
  → Tries: direct → STUN → TURN (in order of preference)
  → Used by WebRTC (Google Meet, Discord, Zoom)
  
ALG   (Application Layer Gateway):
  → Router inspects FTP/SIP payload, rewrites IPs
  → Enables FTP active mode, SIP through NAT
EXAMPLES
1. WhatsApp call (VoIP): uses STUN to discover public IP,
   then ICE to negotiate best path through NAT.

2. FTP through NAT: ALG rewrites PORT command payload
   from private IP to public IP automatically.

3. PS4/Xbox gaming: UPnP auto-creates NAT port forwarding
   entries. "Open NAT" = UPnP working. "Strict NAT" = blocked.

ARP — Address Resolution Protocol

💡 ANALOGY
ARP is like shouting in a room: "Hey, who is John Smith? Please come talk to me!" Everyone hears the question, but only John responds — and then you remember his location.

ARP resolves IP addresses (Layer 3) to MAC addresses (Layer 2) within a local network. Before a device can send a frame, it must know the destination's MAC. ARP broadcasts a request; only the target replies. Results are cached to avoid repeated broadcasts.

ARP request/reply flow
1
ARP Request (Broadcast)
PC1 → FF:FF:FF:FF:FF:FF — "Who has 192.168.1.2? Tell 192.168.1.1"
All devices on segment receive this frame
2
ARP Reply (Unicast)
PC2 → PC1 (unicast) — "192.168.1.2 is at AA:BB:CC:DD:EE:FF"
Only the target with 192.168.1.2 responds
3
Cache Update
PC1 stores: 192.168.1.2 → AA:BB:CC:DD:EE:FF [expires ~20min]
No more broadcasts needed until cache expires
ARP operates at the boundary of Layer 2 and Layer 3. Every time a device wants to send data to an IP address on the same subnet, it first checks its ARP cache. If the MAC isn't cached, it broadcasts an ARP Request. The target replies with its MAC address, which gets cached. Gratuitous ARP: a device announces its own IP:MAC mapping (used on startup, IP change, or failover).
ARP PACKET FORMAT
ARP Request:
  Sender MAC: 00:1A:2B:3C:4D:5E  (known)
  Sender IP:  192.168.1.1         (known)
  Target MAC: 00:00:00:00:00:00   (unknown — what we want!)
  Target IP:  192.168.1.2         (known)

ARP Reply:
  Sender MAC: AA:BB:CC:DD:EE:FF  (target fills this in)
  Sender IP:  192.168.1.2
  Target MAC: 00:1A:2B:3C:4D:5E
  Target IP:  192.168.1.1

Cache TTL: ~20 min (Linux: 30s default, Windows: 2 min)
EXAMPLES
1. arp -a (Windows/Linux): show ARP cache table
   192.168.1.1 at 00:50:56:ab:cd:ef [ether] on eth0

2. Gratuitous ARP: HSRP/VRRP sends when virtual IP
   moves to standby router → all hosts update their cache.

3. ARP for gateway: PC always ARPs for default gateway MAC
   first, then forwards all non-local traffic to that MAC.
Proxy ARP allows a router to answer ARP requests on behalf of hosts on different networks. If Host A (10.0.0.1/8) sends ARP for 10.1.0.1, its router intercepts and replies with its own MAC. Host A sends packets to router thinking it's sending directly — the router then forwards appropriately. Enabled by default on some Cisco platforms; generally considered a legacy feature.
PROXY ARP LOGIC
If router receives ARP Request AND:
  → Target IP is reachable via another interface
  → Router has route to target IP
Then:
  → Router replies with its own MAC address
  → Sender sends frames to router's MAC
  → Router forwards to actual destination

Cisco: ip proxy-arp (interface command, on by default)
EXAMPLES
1. Used when hosts are configured without a default gateway —
   router transparently proxies all off-subnet ARPs.

2. Can mask misconfigurations (hosts not knowing their subnet)
   — generally better to configure proper default gateway.

3. Disable: no ip proxy-arp on router interface for security.
ARP has no authentication. Any device can send a gratuitous ARP with false information, redirecting traffic. An attacker sends fake ARP replies claiming the router's IP maps to the attacker's MAC. All traffic meant for the router flows through the attacker — a Man-in-the-Middle (MITM) attack. Mitigated by Dynamic ARP Inspection (DAI) on managed switches.
ATTACK & DEFENSE
Attack:
  1. Attacker sends: "192.168.1.1 (router) is at ATTACKER_MAC"
  2. All hosts update ARP cache with wrong MAC
  3. Traffic → attacker (who may forward to real router)
  4. Attacker sees all plaintext data (MITM!)

Defense:
  DAI (Dynamic ARP Inspection): validates ARP against
  DHCP snooping table. Invalid ARP packets dropped.
  
  DHCP Snooping: trusted vs untrusted ports; maintains
  IP-MAC-Port binding database for DAI to reference.
EXAMPLES
1. Tools: arpspoof (dsniff), ettercap, Scapy — all exploit
   ARP's lack of authentication to perform MITM.

2. Cisco DAI config:
   ip arp inspection vlan 10
   interface Fa0/0 → ip arp inspection trust (uplink only)

3. HTTPS/TLS protects against MITM even if ARP poisoned —
   certificate mismatch alerts user to tampering.

DHCP — Dynamic Host Configuration Protocol

💡 ANALOGY
DHCP is like a hotel front desk — when you check in (connect to network), they give you a room number (IP address) for your stay (lease time). When you check out, the room becomes available for the next guest.

DHCP automatically assigns IP addresses, subnet masks, default gateways, DNS servers, and other parameters to hosts. Without DHCP, every device requires manual IP configuration — impractical for large networks. DHCP uses UDP port 67 (server) and 68 (client).

DORA — the DHCP handshake
D
DISCOVER (broadcast)
Client → 255.255.255.255 — "I need an IP address! Is anyone a DHCP server?"
Src IP: 0.0.0.0 | Dst IP: 255.255.255.255 | UDP 68→67
O
OFFER (broadcast)
Server → 255.255.255.255 — "I offer you 192.168.1.105/24, gateway .1, DNS 8.8.8.8, lease 24h"
Server temporarily reserves the offered IP
R
REQUEST (broadcast)
Client → 255.255.255.255 — "I accept the offer from server 192.168.1.1 for 192.168.1.105"
Broadcast so other DHCP servers know this offer was accepted
A
ACKNOWLEDGE (broadcast/unicast)
Server → Client — "Confirmed! Here's your full config. Lease expires in 24 hours."
Client can now use the assigned IP address
DHCP leases aren't permanent. T1 (50% of lease time) is when the client sends a unicast DHCPREQUEST to renew with its original server. T2 (87.5%) is when it broadcasts to any available server. If the lease expires with no renewal, the client must release the IP and restart DORA. This ensures IP addresses return to the pool when devices leave.
LEASE TIMER FORMULAS
T1 (renewal timer)  = 0.5   × lease_time  → unicast to server
T2 (rebinding timer) = 0.875 × lease_time  → broadcast

Example: lease = 24 hours (86,400 seconds)
  T1 = 12h  → "Server, may I keep 192.168.1.105?"
  T2 = 21h  → "Any server, may I keep 192.168.1.105?"
  T3 = 24h  → Lease expired → start DORA again

DHCPRELEASE: client explicitly returns IP before expiry
DHCPDECLINE:  client rejects offered IP (e.g., ARP probe 
              detects IP already in use on network)
EXAMPLES
1. Laptop closes lid at T1: can still renew on reconnect
   as long as lease hasn't expired.

2. Server-side show: DHCP pool, leased addresses, expiry times.
   Prevents IP conflicts by tracking all active leases.

3. Short lease (1h) vs long lease (8d): short = reclaims IPs
   faster (good for coffee shops). Long = less broadcast traffic
   (good for stable enterprise desktops).
DHCP uses broadcasts which don't cross routers. DHCP Relay Agents (ip helper-address in Cisco) intercept DHCP broadcasts and forward them as unicast to a central DHCP server. The server uses the relay's IP (giaddr field) to determine which IP pool to assign from. DHCP Options provide additional configuration parameters.
DHCP RELAY & KEY OPTIONS
Cisco relay config:
  interface Vlan10
    ip helper-address 10.0.0.5  ← DHCP server IP

Key DHCP Options (RFC 2132):
  Option 1:  Subnet mask
  Option 3:  Default router/gateway
  Option 6:  DNS server(s)
  Option 15: Domain name
  Option 43: Vendor-specific info
  Option 51: IP address lease time
  Option 53: DHCP message type (Discover/Offer/Request/ACK)
  Option 82: Relay agent information (circuit ID, remote ID)
EXAMPLES
1. Enterprise with 50 VLANs: one Windows DHCP server.
   Each VLAN SVI has ip helper-address → all VLANs served.

2. Option 82 lets server know which switch port/VLAN the
   client is on → assign from correct scope/pool.

3. DHCP starvation attack: attacker sends thousands of
   DISCOVER messages, exhausting IP pool. Defense: DHCP
   snooping with rate-limiting on untrusted ports.
IPv6 has two auto-configuration methods: SLAAC (Stateless Address Autoconfiguration) and DHCPv6. SLAAC uses Router Advertisements (ICMPv6) — hosts generate their own address from the /64 prefix. DHCPv6 provides more control (DNS, domain, specific addresses). Stateless DHCPv6 provides config but not addresses; Stateful DHCPv6 provides both.
SLAAC ADDRESS GENERATION
Prefix from Router Advertisement: 2001:db8::/64
Interface ID from EUI-64:
  MAC: 00:1A:2B:3C:4D:5E
  EUI-64: 021A:2BFF:FE3C:4D5E (insert FFFE, flip U/L bit)
Full address: 2001:db8::021a:2bff:fe3c:4d5e/64

Privacy Extensions (RFC 4941): random interface ID
  → Prevents tracking via stable MAC-derived address
EXAMPLES
1. SLAAC: device auto-configures global address from RA prefix
   + generates link-local fe80::... automatically.

2. Stateless DHCPv6 (M=0,O=1 in RA): SLAAC for address,
   DHCPv6 just for DNS/domain name configuration.

3. Stateful DHCPv6 (M=1 in RA): full DHCPv6 like DHCPv4
   — assigns specific IPv6 addresses from server pool.

ICMP — Internet Control Message Protocol

💡 ANALOGY
ICMP is the postal system's notification service — it sends "return to sender, address unknown" when your letter can't be delivered, or "road closed ahead" when a path is unavailable.

ICMP is a support protocol used for sending error messages and operational queries. It's encapsulated in IP packets but is not used for end-user data transfer. ICMP enables essential diagnostic tools: ping (connectivity test) and traceroute (path discovery).

Each ICMP message has a Type (1 byte) and Code (1 byte) field. Type identifies the category; Code provides detail within that type. The most common types are Echo Request/Reply (ping), Destination Unreachable (various codes for different failure reasons), Time Exceeded (traceroute), and Redirect (inform sender of better route).
COMMON ICMP TYPES & CODES
Type 0:  Echo Reply       (ping response)
Type 3:  Destination Unreachable
  Code 0: Net unreachable    Code 1: Host unreachable
  Code 2: Protocol unreachable Code 3: Port unreachable ← UDP!
  Code 4: Fragmentation needed (DF set, MTU too small)
Type 5:  Redirect (code 0-3: net/host/TOS redirect)
Type 8:  Echo Request      (ping command sends this)
Type 11: Time Exceeded
  Code 0: TTL exceeded in transit    ← traceroute!
  Code 1: Fragment reassembly timeout
Type 12: Parameter Problem (bad IP header)
EXAMPLES
1. "Connection refused" (TCP) vs ICMP Type 3 Code 3:
   UDP to closed port → host sends ICMP Port Unreachable.
   TCP to closed port → TCP RST (no ICMP).

2. Type 3, Code 4 (Frag Needed): enables Path MTU Discovery.
   Router can't forward oversized DF=1 packet → sends this.

3. Type 5 Redirect: router tells PC "use 192.168.1.2 as
   gateway for 10.0.0.0/8, it's more optimal" → PC updates
   its local routing table temporarily.
Ping sends ICMP Type 8 (Echo Request) packets and waits for Type 0 (Echo Reply). Each request has a sequence number and identifier. The RTT (round-trip time) measures network latency. Packet loss percentage indicates link quality. The TTL in the reply reveals approximate hop count.
PING METRICS
RTT (Round Trip Time) = T_reply − T_request
Packet Loss % = ((sent − received) / sent) × 100

ICMP Echo payload includes:
  Identifier (2B): identifies the ping session
  Sequence (2B): increments per request
  Timestamp (optional): for RTT calculation
  Data: arbitrary padding (default 56 bytes = 64 byte ICMP)

ping -c 4 -s 1400 -t 128 192.168.1.1
  -c: count, -s: payload size, -t: TTL to use
EXAMPLES
1. ping 8.8.8.8:
   64 bytes from 8.8.8.8: seq=1 ttl=118 time=13.8ms
   Reply TTL=118 → started at 128 (Windows default)
   → 128−118 = 10 hops away (approx)

2. ping -f -l 1400 192.168.1.1 (Windows): DF=1, 1400 bytes
   If "Packet needs to be fragmented" → MTU < 1420 on path.

3. Ping flood (DoS): ping -f -s 65500 target → overwhelm
   network. Modern routers rate-limit ICMP replies.
Traceroute exploits ICMP's Time Exceeded message. It sends packets with TTL starting at 1, increasing by 1 each round. Each router that decrements TTL to 0 returns an ICMP Time Exceeded, revealing its IP address. By collecting all these responses, traceroute maps the entire path to the destination.
TRACEROUTE ALGORITHM
For TTL = 1, 2, 3, ..., max_hops:
  Send 3 packets with current TTL to destination
  
  Router at hop N:
    → Decrements TTL to 0
    → Discards packet
    → Sends ICMP Time Exceeded (Type 11, Code 0) to source
    
  Source receives ICMP from router_N_IP → that's hop N!
  
  When ICMP Echo Reply (Type 0) received:
    → Destination reached, traceroute complete
  
  * = timeout (router doesn't reply, firewall blocks ICMP)
EXAMPLES
1. traceroute google.com:
   1  192.168.1.1 (home router)     1.3ms
   2  10.20.30.1  (ISP gateway)     8.6ms
   3  72.14.215.1 (Google network)  12.4ms
   4  8.8.8.8     (destination)     13.1ms

2. Windows uses ICMP Echo; Linux/Mac use UDP by default.
   tracert (Windows) vs traceroute (Linux/Mac) vs mtr.

3. * * * at a hop = router drops ICMP TTL exceeded (common
   at ISP backbones for security). Doesn't mean path broken.

Routing Algorithms

💡 ANALOGY
Distance Vector = asking neighbors for directions (trusting them blindly). Link State = having a complete city map. Hierarchical = having detailed local maps with highway summaries for other cities.

Routing algorithms determine the best path for packets to travel from source to destination across an internetwork. They fall into three main categories: Distance Vector (simpler, slower convergence), Link State (faster convergence, more resource-intensive), and Hierarchical (scalable for large networks).

Each router maintains a distance vector: a table of (destination, distance, next-hop). Routers periodically share their entire tables only with directly connected neighbors. Each router updates using the Bellman-Ford equation. Simple to implement but slow to converge and vulnerable to routing loops. RIP uses hop count (max 15; 16 = infinity/unreachable).
BELLMAN-FORD EQUATION
D(x, y) = min over all neighbors v of { cost(x,v) + D(v,y) }

Where:
  D(x,y)   = best known distance from x to y
  cost(x,v) = link cost from x to neighbor v
  D(v,y)   = neighbor v's advertised distance to y

RIP Timers:
  Update:  30s  (send full routing table to neighbors)
  Invalid: 180s (route marked invalid if no update)
  Flush:   240s (route removed from table)
  Holddown: prevents accepting new info for failed route
Distance vector example — router A learning about D
A
──1──
B
"D=2"
──2──
C
"D=4"
──3──
D
A hears from B: D is 2 hops away via B (cost 1+2=3). A hears from C: D is 4 hops via C (cost 1+4+... wait, no direct A-C). Best: via B, total cost 3.
EXAMPLES
1. RIPv1: classful, no subnet mask in updates, broadcast.
   RIPv2: classless (CIDR/VLSM), multicast 224.0.0.9.
   RIPng: RIP for IPv6, multicast FF02::9.

2. show ip rip database (Cisco) — view RIP routing entries.
   debug ip rip — see RIP updates being sent/received.

3. Converge time = up to 3 update intervals (90s for RIP)
   after a topology change. OSPF converges in seconds.
When a link fails in distance vector routing, routers may loop-advertise false paths with ever-increasing metrics until they hit infinity (16 in RIP). This count-to-infinity problem can take minutes to resolve. Solutions: split horizon, poison reverse, triggered updates, and hold-down timers.
SOLUTIONS TO COUNT-TO-INFINITY
Split Horizon:
  Never advertise a route back out the interface
  you received it from. Prevents two-node loops.

Poison Reverse (enhanced split horizon):
  Advertise failed routes back with metric=infinity(16)
  More aggressive, eliminates loops faster.

Triggered Updates:
  Send update IMMEDIATELY when route changes.
  Don't wait for 30s periodic timer. Faster convergence.

Hold-Down Timer:
  After hearing a route is gone, ignore updates about
  that route for hold-down period (prevents false info).
  
Route Poisoning:
  Immediately set failed route to metric=16, advertise.
EXAMPLES
1. A-B-C topology. B-C link fails:
   Without split horizon: A tells B "I can reach C via 2 hops"
   → B thinks C reachable via A at cost 3 → A updates to 4...
   → Count-to-infinity!  Resolved only after metric hits 16.

2. With split horizon: B learned C from B-C link. A doesn't
   re-advertise C back to B (learned from B). Loop prevented!

3. Hold-down = 180s in RIP. After route down, ignore any
   updates claiming that route is back for 180s.
Each router floods its Link State Advertisement (LSA) containing its directly connected neighbors and link costs to ALL routers in the area. Every router builds an identical Link State Database (LSDB — complete topology map) and independently runs Dijkstra's Shortest Path First algorithm to calculate optimal routes. Faster convergence, no count-to-infinity, but requires more memory and CPU.
DIJKSTRA'S ALGORITHM
Initialize: dist[source]=0, dist[all others]=∞
Unvisited = all nodes

While unvisited not empty:
  u = node with minimum dist[] in unvisited
  Mark u as visited
  For each neighbor v of u:
    new_dist = dist[u] + cost(u, v)
    If new_dist < dist[v]:
      dist[v] = new_dist
      prev[v] = u  ← remember path

OSPF cost = Reference bandwidth / Interface bandwidth
  Default ref. BW = 100 Mbps
  FastEthernet (100M) cost = 100/100 = 1
  T1 (1.544M) cost = 100/1.544 ≈ 64
EXAMPLES
1. OSPF LSA types:
   Type 1: Router LSA (neighbors + costs)
   Type 2: Network LSA (multi-access segments)
   Type 3: Summary LSA (inter-area, from ABR)
   Type 5: External LSA (redistributed routes, from ASBR)

2. OSPF hello packets: every 10s (default), Dead interval=40s.
   Neighbors form adjacency → exchange LSDB → run Dijkstra.

3. OSPF areas: all areas connect to Area 0 (backbone).
   ABR (Area Border Router) summarizes inter-area routes.
   Reduces LSDB size within each area.
For very large networks, flat routing is impractical — too many routers to track. Hierarchical routing divides the network into regions/areas. Routers know full topology within their region, only summarized information about other regions. OSPF implements this with areas. BGP (Border Gateway Protocol) implements hierarchical routing between Autonomous Systems (ASes) on the internet.
HIERARCHICAL ROUTING MATH
Flat routing (N routers): routing table size = O(N)
  Each router knows every other router's state
  With N=50,000: impractical!

Hierarchical (regions of k routers, N/k regions):
  Intra-region entries = k
  Inter-region entries = N/k
  Total = k + N/k
  Optimal k = √N  (minimizes total entries)
  For N=10,000: optimal k = 100 regions of 100 routers each

BGP (internet routing):
  Path Vector protocol (extends distance vector)
  Propagates entire AS path: prevents loops
  Policy-based: not just shortest path (business decisions)
EXAMPLES
1. OSPF hierarchy: Area 0 backbone ← Area 1,2,3 attached.
   Routers in Area 1 only know Area 1 topology in detail.
   Inter-area: ABR advertises summary LSA3 to other areas.

2. BGP AS path: 192.0.2.0/24 reachable via path AS1, AS3, AS7
   If AS1 sees its own ASN in path → loop detected → discard.

3. Internet BGP: ~70,000 ASes, ~1 million prefixes.
   ISPs filter: customer routes accepted, peer routes filtered
   by prefix lists, AS path filters, and community strings.
